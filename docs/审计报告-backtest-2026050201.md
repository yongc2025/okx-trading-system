## 三、代码质量审计

### 3.1 问题清单

#### 🔴 严重（P0）

**S-01: downloader.py 全局下载器实例 — 线程安全问题**

```python
# api/app.py
_kline_downloader = None  # 全局变量
_order_downloader = None

@app.post("/api/kline/download")
async def api_kline_download(request: Request):
    global _kline_downloader
    ...
    _kline_downloader = KlineDownloader(client)  # 覆盖全局实例
```

**问题**：多个用户同时触发下载会互相覆盖 `_kline_downloader`，导致进度查询返回错误结果，甚至正在运行的下载任务丢失。

**建议**：使用 `dict[str, KlineDownloader]` 按 session/task_id 隔离实例，或限制为单任务模式（已有任务运行时拒绝新请求）。

---

**S-02: API 密钥通过 HTTP Body 明文传输**

```python
@app.post("/api/okx/test")
async def api_okx_test(request: Request):
    body = await request.json()
    client = OKXClient(
        api_key=body.get("api_key", ""),  # 密钥通过 JSON body 传输
        secret=body.get("secret", ""),
        passphrase=body.get("passphrase", ""),
    )
```

**问题**：
1. 密钥通过 HTTP 明文传输（若未配置 TLS）
2. 密钥未持久化，每次刷新页面需重新输入
3. 前端 JS 中密钥可被 XSS 窃取

**建议**：
- 支持通过环境变量或配置文件加载密钥（后端持久化）
- 前端仅做连接测试，不存储密钥
- 生产环境强制 HTTPS

**S-03: SQL 注入风险 — `get_scan_results` 的 `order_by` 参数**

```python
# analysis/extreme_scan.py
def get_scan_results(..., order_by: str = 'change_pct', ...):
    query += f" ORDER BY {order_by} {order_dir} LIMIT ?"  # 直接拼接
```

**问题**：`order_by` 来自前端查询参数，直接拼接到 SQL 中，存在 SQL 注入风险。

**建议**：白名单校验 `order_by`：
```python
VALID_ORDER_FIELDS = {'change_pct', 'scan_time', 'volume', 'symbol'}
if order_by not in VALID_ORDER_FIELDS:
    order_by = 'change_pct'
```

**M-02: 下载器 `_download_single` 分页逻辑缺陷**

```python
# downloader.py KlineDownloader._download_single
while True:
    klines = await self.client.get_history_candles(
        symbol=symbol, bar=bar, limit=DOWNLOAD_BATCH_SIZE,
        after=str(after_ms) if after_ms else None,
    )
    if not klines:
        break
    # 过滤：只保留 >= effective_start 的数据
    filtered = []
    stop = False
    for k in klines:
        ts = int(k[0])
        if ts < effective_start:
            stop = True
            break
        if ts <= end_time:
            filtered.append(k)
    all_klines.extend(filtered)
    ...
    after_ms = int(klines[-1][0])  # 取本页最后一条
```

**问题**：当 `effective_start` 距离当前较远时，前几页数据全被过滤掉但循环继续，可能导致大量无用请求。OKX `history-candles` 返回的数据按时间倒序排列（最新在前），`after` 参数表示"返回此时间戳之前的数据"。如果某页所有数据都 `< effective_start`，应该立即停止而不是继续翻页。

**建议**：在过滤循环中，如果整页数据都 `< effective_start`，直接 `break`。

# data/import_data.py
def import_klines_to_db(kline_data: dict, db_path: str = None):

**M-04: OrderDownloader._pair_fills_to_trades — 未处理 "net" 持仓模式**

```python
if not pos_side or pos_side == "net":
    # 单向持仓模式，根据 side 推断
    if side == "buy":
        direction = "long"
        is_open = True
    else:
        direction = "long"
        is_open = False
```

**问题**：单向持仓模式下，`sell` 被统一当作 `long` 平仓，但实际上 `sell` 也可能是 `short` 开仓（在单向持仓模式下不存在这个组合，但代码没有区分 `buy` 平空仓的情况）。对于 OKX 的单向持仓模式，`buy` 可能是开多或平空，`sell` 可能是开空或平多。

**建议**：对于 `net` 模式，需要根据当前持仓方向来判断是开仓还是平仓，或者直接跳过无法确定的订单并记录警告。

**M-05: BatchSimulator.run — 性能瓶颈**

```python
# analysis/simulator.py
for _, row in closed.iterrows():
    ...
    klines = load_klines(symbol, self.kline_bar, start=entry_time, source="auto")
    ...
```

**问题**：每笔交易都单独调用 `load_klines()`，如果同一个币种有 N 笔交易，就会查询数据库 N 次。对于 500 笔交易，这是严重的 N+1 查询问题。

**建议**：
1. 预加载所有涉及币种的 K 线数据到内存
2. 按 symbol 分组，每个 symbol 只查一次
3. 使用 `load_all_klines_from_db(bar)` 一次加载

## 五、性能审计

### 5.1 性能瓶颈清单

| 编号 | 问题 | 位置 | 影响 | 建议 |
|------|------|------|------|------|
| PERF-01 | N+1 查询（模拟器逐笔加载 K 线） | `simulator.py` BatchSimulator.run | 500 笔交易 = 500 次 DB 查询 | 预加载到内存 |

| PERF-06 | pkl 全量加载 | `loader.py` load_all_pkl | 每次扫描加载所有 pkl | 按需加载单个 symbol |

|----|----------|----------|------|

| `kline_data` | (symbol, bar), time | — | ✅ 充分 |

| `scan_results` | symbol, scan_time, change_pct | direction | 🟡 可加 |

### 5.3 内存使用分析

| 场景 | 估计内存 | 风险 |
|------|----------|------|
| 加载 20 个币种 × 90 天 1H K 线 | ~50MB | ✅ 安全 |
| 加载 20 个币种 × 90 天 1m K 线 | ~3GB | 🔴 危险 |
| 500 笔交易的模拟回测 | ~10MB | ✅ 安全 |
| 参数网格搜索 (4×5=20 组) | ~200MB | 🟡 注意 |

**建议**：1m K 线数据量巨大，应避免全量加载到内存。极端行情扫描应改为流式处理或按 symbol 逐个加载。

def import_klines_to_db(kline_data: dict, db_path: str = None):
    from backtest.data.schema import get_connection, TABLE_KLINE_DATA
    conn = get_connection(db_path)
    for symbol, df in kline_data.items():
        for _, row in df.iterrows():
            conn.execute(
                f"INSERT OR IGNORE INTO {TABLE_KLINE_DATA} "
                "(symbol, bar, time, open, high, low, close, volume, amount, source) "
                "VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)",
                (symbol, '1H', str(row['time']),
                 row['open'], row['high'], row['low'], row['close'],
                 row.get('volume'), row.get('amount'), 'sample')
            )
    conn.commit()
    conn.close()

## 十、总结

### 做得好的地方

1. **功能覆盖全面** — 需求文档中 7 个阶段、10 个页面、22 个 API 端点全部实现
2. **模块划分清晰** — data / analysis / api / templates 四层架构，职责明确
3. **代码注释充分** — 中文注释、docstring 覆盖率高
4. **数据源抽象统一** — loader.py 的 auto/db/pkl 三模式设计优雅
5. **前端风格统一** — 暗色主题、ECharts 图表、响应式布局一致
6. **错误处理基本到位** — 大部分函数有 try/except 和 fallback

### 需要改进的地方

1. **安全性** — SQL 注入和密钥管理需要加固
2. **性能** — N+1 查询和全量加载是主要瓶颈
3. **代码重复** — database.py 两份实现需要合并
4. **测试覆盖** — 新增模块（downloader, simulator, order_analysis）缺少单元测试
5. **并发安全** — 全局下载器实例设计有缺陷
6. **前端健壮性** — showToast 未定义、无分页、无 loading 状态处理

### 修复优先级

```
P0 (必须): SQL注入 + 线程安全 + N+1查询  → 预计 2-3 小时
P1 (建议): 密钥管理 + 代码重复 + CORS + 边界修复  → 预计 3-4 小时
P2 (可选): 日志级别 + 随机种子 + 分页 + 连接池  → 预计 2-3 小时
```
