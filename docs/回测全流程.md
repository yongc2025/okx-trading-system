## 三、阶段 1 — 数据采集

### 3.1 K 线下载

**目标**：用户在前端选择币种和 K 线周期，从 OKX 拉取历史 K 线数据，存入 SQLite。

**数据源**：OKX REST API `/api/v5/market/history-candles`

**已有基础**：
- 需在 `backtest/data/okx_client.py` 新建轻量 OKX 客户端（独立于 trading 模块，回测可单独运行）

> **为什么不复用 `trading/api/okx_rest.py`**：
> - trading 模块有自己独立的配置、日志、凭证管理、代理设置
> - 回测模块需要独立运行，不能依赖 trading 模块的环境
> - trading 的客户端包含大量下单/持仓/止损接口，回测用不上
> - 解耦后两个模块可独立部署、独立升级

| 历史订单提取 | 从已下载的订单中提取涉及的币种，自动填入选择 |

#### K 线周期

| 周期 | OKX 参数 | 90天数据量/币种 | 用途 |
|---|---|---|---|
| 1 分钟 | `1m` | ~129,600 条 | 极端行情扫描（已有） |
| 5 分钟 | `5m` | ~25,920 条 | 止盈止损模拟（推荐默认） |
| 15 分钟 | `15m` | ~8,640 条 | 中线分析 |
| 1 小时 | `1H` | ~2,160 条 | 趋势判断 |
| 4 小时 | `4H` | ~540 条 | 宏观行情 |
| 1 天 | `1D` | ~90 条 | 长线参考 |

**用户可同时选择多个周期**，系统按 `(symbol, bar)` 维度并行下载，每个维度独立追踪进度和断点。

#### 下载策略

- 按 `(symbol, bar)` 维度逐个下载，每个维度独立维护进度
- OKX 限速：20 次 / 2 秒，需要控制请求频率（跨维度共享限速计数器）
- 单次最多返回 300 条 K 线，需要分页循环拉取
- 增量下载：记录每个 `(symbol, bar)` 的最新时间戳，只拉取增量
- 断点续传：下载中断后从 `download_status` 表记录的位置继续
- 前端通过轮询 `/api/data/progress` 获取实时进度

### 3.2 订单下载

**目标**：从 OKX 拉取历史成交记录，清洗后存入 `trade_records` 表。

**数据源**：OKX REST API `/api/v5/trade/fills-history`

**已有基础**：
- 需在 `backtest/data/okx_client.py` 中实现 `get_fills_history()` 接口

**OKX 字段 → 本地字段映射**：

| OKX 字段 | 本地字段 | 说明 |
|---|---|---|
| `instId` | `symbol` | 币种，如 BTC-USDT-SWAP |
| `side` + `posSide` | `direction` | buy+long=做多开仓, sell+long=做多平仓 |
| `fillPx` | `entry_price` / `exit_price` | 成交价格 |
| `fillSz` | `entry_qty` / `exit_qty` | 成交数量 |
| `fee` | 需计算 | 手续费（负数=返佣） |
| `ts` | `entry_time` / `exit_time` | 成交时间戳 |
| `ordId` | `trade_id` | 关联同一订单的多笔成交 |
| `posSide` | `direction` | long / short |

**数据清洗流程**：
```
1. 拉取原始 fills 数据
2. 按 ordId 聚合同一订单的多笔成交（加权平均价格、总量）
3. 按 symbol + direction 配对开仓和平仓订单
4. 计算每笔完整交易的指标：
   - pnl = (exit_price - entry_price) × qty × direction_factor
   - entry_cost = entry_price × qty / leverage
   - roi = pnl / entry_cost
   - 持仓时长 = exit_time - entry_time
5. 写入 trade_records 表
```

**分页策略**：
- OKX 单次最多返回 100 条
- 使用 `after` 参数做时间游标，循环拉取直到无更多数据
- 记录拉取进度，支持断点续传

### 3.3 订单导入（CSV）

**目标**：支持用户手动导入交易记录（无需 API 权限）。

**适用场景**：
- 用户没有 OKX API 权限
- 导入其他平台/工具的交易记录
- 导入别人分享的交易数据做分析

#### 导入模板

**最小字段模板**（用户只需填 6 个字段，其余自动计算）：

```csv
symbol,direction,entry_time,entry_price,exit_time,exit_price
BTC-USDT-SWAP,long,2025-01-15 10:30,42500.5,2025-01-15 14:20,43100.2
ETH-USDT-SWAP,short,2025-01-16 09:00,2650.0,2025-01-16 18:30,2580.5
```

| 字段 | 必填 | 说明 |
|---|---|---|
| `symbol` | ✅ | 币种，如 BTC-USDT-SWAP |
| `direction` | ✅ | long / short |
| `entry_time` | ✅ | 开仓时间，支持 `YYYY-MM-DD HH:MM` 格式 |
| `entry_price` | ✅ | 开仓价格 |
| `exit_time` | ✅ | 平仓时间 |
| `exit_price` | ✅ | 平仓价格 |

**可选字段**（不填则自动计算或使用默认值）：

| 字段 | 说明 | 默认值 |
|---|---|---|
| `leverage` | 杠杆倍数 | 1 |
| `entry_qty` | 开仓数量 | 按 entry_cost 反算 |
| `entry_cost` | 开仓成本 (USDT) | 1000 |
| `position_tier` | 仓位档位 | first |

**自动计算字段**（用户不需要填）：
```
exit_value  = entry_qty × exit_price / leverage
pnl         = (exit_price - entry_price) × entry_qty × direction_factor
pnl_rate    = pnl / account_capital
roi         = pnl / entry_cost
is_win      = pnl > 0 ? 1 : 0
is_loss     = pnl < 0 ? 1 : 0
```


#### 数据校验

导入时自动校验：
- `symbol` 格式是否合法
- `direction` 是否为 long/short
- `entry_time` < `exit_time`
- `entry_price` / `exit_price` > 0
- 重复数据检测（symbol + entry_time 去重）

异常数据不导入，返回错误报告供用户修正。

### 3.4 数据源选择与格式统一

**目标**：用户可选择 K 线数据来源（交易所下载 / 本地 pkl），回测引擎不感知数据来源差异。

#### 数据源选项

| 来源 | 入口 | 格式 | 说明 |
|---|---|---|---|
| OKX 交易所 | 前端触发下载 | OKX API 返回格式 | 最权威，支持增量更新 |
| 本地 pkl 文件 | 上传到 `data/pkl/` | 邢不行框架格式 / 单 DataFrame | 已有历史数据直接用 |
| 样例数据 | 系统自动生成 | 模拟数据 | 无真实数据时测试用 |

#### 格式差异

| 字段 | OKX API 返回 | pkl (邢不行格式) | 统一后 |
|---|---|---|---|
| 时间 | `ts` (毫秒时间戳) | `candle_begin_time` / `date` | `time` (ISO 8601) |
| 开盘 | `o` | `open` / `Open` | `open` |
| 最高 | `h` | `high` / `High` | `high` |
| 最低 | `l` | `low` / `Low` | `low` |
| 收盘 | `c` | `close` / `Close` | `close` |
| 成交量 | `vol` | `volume` / `VOL` / `vol` | `volume` |
| 成交额 | `volCcy` | `amount` / `quote_vol` | `amount` |

#### 统一处理流程

```
数据来源
  ├── OKX API    → okx_client.py 内部转换 → 标准格式
  ├── pkl 文件   → loader.py _standardize_columns() → 标准格式（已有）
  └── 样例生成   → generate_sample.py 直接生成标准格式（已有）

标准格式 → kline_data 表（SQLite）或内存 DataFrame
                ↓
         分析模块（不感知来源）
```

**关键设计**：`loader.py` 作为统一数据入口，分析模块只调用 loader，不直接接触数据源。

```python
# 统一接口（新增）
def load_klines(symbol, bar, start=None, end=None, source="auto"):
    """
    source:
      "db"    → 从 SQLite kline_data 表读取
      "pkl"   → 从 pkl 文件读取
      "auto"  → 优先 DB，fallback 到 pkl
    """
```

### 4.1 数据库结构

**数据库文件**：`backtest/data/backtest.db`（SQLite，WAL 模式）

#### 已有表

**trade_records — 交易记录表**

```sql
CREATE TABLE trade_records (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    trade_id        TEXT UNIQUE,              -- 交易唯一标识
    symbol          TEXT NOT NULL,            -- 币种 (如 BTC-USDT-SWAP)
    direction       TEXT NOT NULL,            -- long / short
    leverage        INTEGER DEFAULT 1,        -- 杠杆倍数
    position_tier   TEXT,                     -- first / add1 / add2
    entry_time      TEXT NOT NULL,            -- 开仓时间 (ISO 8601)
    entry_price     REAL NOT NULL,            -- 开仓价格
    entry_qty       REAL NOT NULL,            -- 开仓数量
    entry_cost      REAL NOT NULL,            -- 开仓成本 (USDT)
    exit_time       TEXT,                     -- 平仓时间
    exit_price      REAL,                     -- 平仓价格
    exit_qty        REAL,                     -- 平仓数量
    exit_value      REAL,                     -- 平仓价值
    pnl             REAL,                     -- 盈亏金额 (USDT)
    pnl_rate        REAL,                     -- 盈亏比率 (相对总资金)
    roi             REAL,                     -- 投资回报率 (相对本金)
    is_win          INTEGER,                  -- 1=盈利 0=非盈利
    is_loss         INTEGER,                  -- 1=亏损 0=非亏损
    max_floating_loss       REAL DEFAULT 0,   -- 最大浮亏金额
    max_floating_loss_rate  REAL DEFAULT 0,   -- 最大浮亏比率
    exceeded_stoploss       INTEGER DEFAULT 0,-- 是否超过止损线
    account_capital REAL,                     -- 当时账户总资金
    created_at      TEXT DEFAULT (datetime('now'))
);
-- 索引: symbol, entry_time, direction
```

**position_snapshots — 持仓快照表**

```sql
CREATE TABLE position_snapshots (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    trade_id        TEXT NOT NULL,            -- 关联 trade_records.trade_id
    symbol          TEXT NOT NULL,
    snapshot_time   TEXT NOT NULL,
    price           REAL NOT NULL,            -- 快照时价格
    floating_pnl    REAL NOT NULL,            -- 浮盈浮亏
    floating_pnl_rate REAL NOT NULL,          -- 浮盈浮亏比率
    FOREIGN KEY (trade_id) REFERENCES trade_records(trade_id)
);
-- 索引: trade_id, snapshot_time
```

**scan_results — 极端行情扫描结果表**

```sql
CREATE TABLE scan_results (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,

    high_price      REAL,

    volume          REAL,

);
-- 索引: symbol, scan_time, change_pct
```

**kline_data — K 线数据表**

```sql
CREATE TABLE kline_data (
```

## 七、阶段 5 — 止盈止损模拟（核心改造）

### 7.1 现状与问题

**当前做法**（`stoploss_sim.py`）：
- 只做止损模拟，没有止盈
- 逻辑是"如果亏损超过 X%，就截断为 X%"
- 不考虑实际价格触发点、触发时间、K 线走势

**问题**：
- 模拟结果过于理想化（假设精确在止损比例平仓）
- 没有考虑同一根 K 线内价格波动（先触发止损还是止盈）
- 无法回答"如果当时设了止盈 X%，结果会怎样"

### 7.2 改造方案 — K 线逐 bar 回放

**核心思路**：对每笔历史订单，找到其持仓期间的 K 线数据，逐 bar 检测止盈止损触发。

**算法流程**：

```
输入:
  - 订单: entry_time, entry_price, direction, leverage, entry_cost
  - 参数: stoploss_pct, takeprofit_pct
  - K 线: 该币种在 [entry_time, ...] 期间的 bars

处理:
  1. 根据 entry_price 和参数计算触发价:
     if direction == "long":
       stoploss_price   = entry_price × (1 - stoploss_pct)
       takeprofit_price = entry_price × (1 + takeprofit_pct)
     if direction == "short":
       stoploss_price   = entry_price × (1 + stoploss_pct)
       takeprofit_price = entry_price × (1 - takeprofit_pct)

  2. 从 entry_time 开始逐 bar 遍历:
     for bar in klines[entry_time:]:
       if direction == "long":
         if bar.low <= stoploss_price:
           → 触发止损, 平仓价 = stoploss_price (或 bar.open 取较低者)
           退出循环
         if bar.high >= takeprofit_price:
           → 触发止盈, 平仓价 = takeprofit_price (或 bar.open 取较高者)
           退出循环
       if direction == "short":
         if bar.high >= stoploss_price:
           → 触发止损
           退出循环
         if bar.low <= takeprofit_price:
           → 触发止盈
           退出循环

  3. 如果遍历完所有 bar 都没触发:
     → 按最后一根 bar 的 close 强制平仓（持仓超时）

  4. 计算模拟结果:
     simulated_pnl = (exit_price - entry_price) × qty × direction_factor
     hold_bars = 触发 bar 的 index - 开仓 bar 的 index
     trigger_type = "stoploss" / "takeprofit" / "timeout"
```

**同一根 K 线内同时触及止盈止损的处理**：
```
默认策略（保守）: 假设先触发止损
可配置策略:
  - "stoploss_first"  → 保守（默认）
  - "takeprofit_first" → 乐观
  - "open_compare"     → 用 open 价格推断方向
```

### 7.3 多参数对比矩阵

```
               止损比例
         │  5%  │ 10%  │ 15%  │ 20%  │ 不设 │
─────────┼──────┼──────┼──────┼──────┼──────┤
止  5%   │  ○   │  ○   │  ○   │  ○   │  ○   │
盈 10%   │  ○   │  ○   │  ○   │  ○   │  ○   │
比 20%   │  ○   │  ○   │  ○   │  ○   │  ○   │
例 50%   │  ○   │  ○   │  ○   │  ○   │  ○   │
  不设   │  ○   │  ○   │  ○   │  ○   │  ○   │

每个 ○ = 一组完整模拟结果:
  - 胜率、盈亏比、总收益、最大回撤
  - 触发类型分布（止盈/止损/超时各占多少）
  - 平均持仓时长
  - 收益曲线数据
```

### 7.4 滑点处理

```
可配置滑点比例 (默认 0 = 不加):
  止损滑点: 实际平仓价 = stoploss_price × (1 - slippage)
  止盈滑点: 实际平仓价 = takeprofit_price × (1 - slippage)
```

### 7.5 对外接口（保持不变）

