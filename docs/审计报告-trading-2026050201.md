## 一、结论摘要

两份报告都指出了 `trading` 模块存在明显问题，但侧重点不同：

- 第三方报告更强在：
  - 静态代码问题
  - 代码卫生
  - 安全细节
  - 缺失测试项
- 本地审计更强在：
  - 需求闭环
  - 前后端一致性
  - 交易链路真实性
  - 实盘交付风险

最终采纳原则：

- 用本地审计作为 `trading` 模块的主判断依据
- 吸收第三方报告中成立的代码级和安全级缺陷

最终评级：

- 功能完成度：`B-`
- 交易安全性：`C`
- 前端闭环程度：`C+`
- 可交付性：`C`

**结论：当前 `trading` 模块不建议直接用于高风险实盘交付。**

## 五、本地审计更关键的新增结论

这些问题第三方报告没有充分强调，但它们更直接决定 `trading` 模块能否安全交付。

### 5.1 下单成功判定过宽

当前下单流程可能出现：

1. OKX 子单失败
2. 本地仍写入 `trade_records`
3. API 层继续挂止损

这会导致：

- 本地伪成交
- 伪止损
- 审计记录失真

最终采纳结论：

- `P0`

代码依据：
[trading/engine/order.py](/D:/workspace/okx-trading-system/trading/engine/order.py:79)
[trading/api/app.py](/D:/workspace/okx-trading-system/trading/api/app.py:326)

### 5.2 前端数量计算模型失真

前端当前用固定：

`contractVal = 10`

来估算数量。

这会影响：

- 下单数量
- 名义价值
- 拆单阈值
- 风控判断

如果合约面值不是这个值，计算就整体失真。

最终采纳结论：

- `P0`

代码依据：
[trading/templates/index.html](/D:/workspace/okx-trading-system/trading/templates/index.html:1671)

### 5.3 前端档位与后端风控脱节

前端允许选择：

- `first`
- `add1`
- `add2`

但主下单 API 固定按 `"first"` 校验。

最终采纳结论：

- `P0`

代码依据：
[trading/templates/index.html](/D:/workspace/okx-trading-system/trading/templates/index.html:1418)
[trading/api/app.py](/D:/workspace/okx-trading-system/trading/api/app.py:319)

### 5.4 杠杆没有真实下发到交易所

虽然有 `set_leverage()` 封装，但没看到主交易路径实际调用。

最终采纳结论：

- `P0`

代码依据：
[trading/api/okx_rest.py](/D:/workspace/okx-trading-system/trading/api/okx_rest.py:227)

### 5.5 本地成交生命周期未收敛

平仓后没有看到完整更新：

- `status=closed`
- `pnl`
- `closed_at`

这会直接影响：

- 交易复盘
- 本地审计
- 盈亏看板

最终采纳结论：

- `P0`

### 5.6 加仓前端未闭环

前端没有独立走 `/api/add-position` 的清晰入口。

最终采纳结论：

- `P1`

### 5.7 测试不能稳定执行

当前测试存在：

- Windows 控制台编码问题
- 日志文件初始化问题
- `PYTHONPATH` 启动方式脆弱

最终采纳结论：

- `P0/P1` 之间，建议按 `P1+` 处理
